# Auditing design systems

Install a live `/style-guide` audit page into a React + Tailwind project. The page reads token values directly from rendered CSS in both themes (via `getComputedStyle`), so it cannot drift from the app's real styles. v1 covers colors + a WCAG contrast matrix; the engine and config also carry the seams for cross-source drift (Style Dictionary, Figma, Storybook) added later.

This skill uses the `@agentic-design-audit/core` (engine) and `@agentic-design-audit/react` (`<DesignAudit>`) packages. The config that drives them is generated by a deterministic script — do not hand-write token lists.

## Workflow

Copy this checklist and check items off:

```
- [ ] 1. Detect framework + styling + theme mechanism
- [ ] 2. Generate config from the project's CSS (run extract-tokens.mjs)
- [ ] 3. Review/repair the config (theme scopes, legacy palette)
- [ ] 4. Mount the audit route
- [ ] 5. Verify (route compiles, returns 200, swatches + contrast render)
```

### 1. Detect

Confirm React + Tailwind (this skill's v1 scope). Identify:
- **Framework flavor** → routing target. See [reference/next.md](reference/next.md) (App Router / Pages) or [reference/vite.md](reference/vite.md) (Vite / CRA / React Router).
- **Token source CSS** — the file declaring `--…` custom properties (commonly `globals.css`, `index.css`, `app.css`).
- **Theme mechanism** — class (`.light`/`.dark`) vs attribute (`[data-theme="…"]`) vs media query. This is the most error-prone variable; the extractor reports it and warns when ambiguous.

### 2. Generate config

Run the bundled extractor against the token CSS:

```bash
node scripts/extract-tokens.mjs path/to/globals.css > design-audit.config.json
```

It emits `{ themeScopes, tokenDefs, contrast }`. `tokenDefs` are derived from custom properties (`--color-bg-card` → `color.bg.card`, type `color`); `contrast` pairs every `color.text*` against every `color.bg*` surface.

### 3. Review/repair the config

- **Heed `_warnings`.** If the project forces dark mode only via `:root` (no `.dark` rule), side-by-side comparison needs a scoped dark selector. Add one rule so a subtree can be forced dark:
  ```css
  /* change `:root {` to also match .dark */
  :root, .dark, [data-theme="dark"] { /* …dark tokens… */ }
  ```
  Then ensure the dark `themeScope` uses the matching `className`/`attributes`.
- **Optional enrichments (not derivable from CSS variables).** Add these to the config by hand when the project has them — sections render only when their data is present:
  - `typography: { scale, weights, letterSpacing }` — type scale/weights usually live in a Tailwind config, not CSS variables. `scale` items are `{ label, value }` (a CSS font-size). Font *families* are auto-detected from `--font-*` tokens; these three are the extras.
  - `components: [{ label, items: [{ label, className, tag? }] }]` — project component classes (e.g. `.btn`, `.card`) rendered live in both themes. `tag` defaults to `button`.
  - `legacy: [{ className, value, note }]` — superseded palette (e.g. Tailwind `studio-*`) flagged for removal. The extractor does not parse JS config, so add these manually.

  Keep the generated `tokenDefs`/`contrast` in the JSON file; merge enrichments separately (see `examples/vite-react/src/enrichments.js` + `main.jsx`) so regenerating tokens never clobbers them.

### 4. Mount the route

Render `<DesignAudit config={config} />` at an unlisted route with `noindex`. The component is the only UI; pass the generated config. Routing specifics per framework are in the reference files above.

```jsx
import { DesignAudit } from '@agentic-design-audit/react'
import config from './design-audit.config.json'

export default function StyleGuide() {
  return <DesignAudit config={config} title="Style Guide" />
}
```

### 5. Verify

Run a validator loop — fix and repeat until all pass:
1. Route compiles and returns 200.
2. Swatches show real hex values (not `—`) in both theme columns — if all `—`, the theme `className`/`attributes` don't match the project's actual theme selectors (return to step 3).
3. The contrast matrix renders ratios; any `Fail`/`AA Large` cell is a real audit finding to report to the user.

## Config shape

```jsonc
{
  "themeScopes": [
    { "name": "light", "className": "light" },
    { "name": "dark",  "className": "dark" }      // or "attributes": { "data-theme": "dark" }
  ],
  "tokenDefs": [
    { "id": "color.bg.card", "cssVar": "--color-bg-card", "type": "color", "group": "color.bg" }
  ],
  "contrast": { "text": ["color.text.primary"], "surfaces": ["color.bg.base"] },

  // --- optional enrichments (added by hand; merged over the generated config) ---
  "typography": {
    "scale": [{ "label": "lg", "value": "1.25rem" }],
    "weights": [{ "label": "bold", "weight": 700 }],
    "letterSpacing": [{ "label": "wide", "value": "0.05em" }]
  },
  "components": [
    { "label": ".btn", "items": [{ "label": "Primary", "className": "btn" }] }
  ],
  "legacy": [ { "className": "studio-black", "value": "#ffffff", "note": "named black, renders white" } ]
}
```

`type` is one of `color | dimension | shadow | duration | fontFamily | other` and drives value normalization in the engine.
